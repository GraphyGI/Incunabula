Part II — Root cause diagnosis

Root cause 1 — Elementor JavaScript and CSS overhead High severity
Elementor injects ~400–600 KB of concatenated CSS and multiple JS bundles regardless of page content. Its frontend editor scripts partially load even in production. This alone accounts for an estimated 300–400ms of TBT. This is not solvable by caching alone — cached Elementor payloads still block main thread on first parse. Partially mitigated with Elementor's own "Improved Asset Loading" (beta) but evidence shows the site is not using this.
Root cause 2 — Three competing carousel libraries High severity
Swiper, Slick, and OWL Carousel are all loading simultaneously. Each is ~70–120 KB minified + gzipped. Combined: ~250 KB of carousel JS serving redundant functionality. This likely stems from multiple purchased theme components and separate WooCommerce plugins each bundling their own carousel dependency with no deduplication. Direct contributor to the 1.0s TBT.
Root cause 3 — Hostinger shared hosting, no server-side caching High severity
No page cache, no object cache (Redis/Memcached), no opcode cache confirmation. With 4,000 WooCommerce SKUs, every uncached product page triggers multiple MySQL queries including wp_posts, wp_postmeta, wp_terms, and wc_order lookups. Hostinger shared hosting imposes hard TTFB ceilings (~600–900ms server response time) that cannot be overcome with front-end optimization alone.
Root cause 4 — Unoptimized images Medium severity
The LCP of 7.3s strongly suggests hero and product images are serving full-resolution JPEG/PNG without WebP conversion, responsive srcsets, or lazy loading below the fold. With a book inventory this size, images are likely uploaded at varying resolutions without standardization. This is fixable without structural changes but is a major LCP driver.
Root cause 5 — Jetpack plugin Medium severity
Jetpack loads numerous modules regardless of which features are enabled. Known to add 50–150ms latency through its external API calls to wordpress.com servers on page load. For a site with already-critical TBT, this is non-trivial overhead. Most Jetpack features can be replaced with lighter alternatives.
Realistic optimization ceiling with current stack
With WP Rocket + Cloudflare + image CDN + JS deferral: Mobile score → 55–68 estimated. LCP → 3.0–4.0s (still amber). TBT → 300–500ms (still amber). These numbers are commercially viable but fall short of green scores. The hard ceiling is Elementor's CSS/JS payload on first load. Without replacing Elementor, scores above ~70 mobile are unlikely without heroic engineering effort.
23 documented UX defects (from PDF analysis)
The issues span navigation, product discovery, checkout, mobile, security, and trust. Notably: misaligned header elements, zero-value badge displays, invasive promotional banners blocking WhatsApp CTA, overlapping login/register forms, plain-text passwords in emails, English copy remaining in checkout, ambiguous "Nuevo" label, no "Buy Now" CTA, and no related products engine. None of these are WooCommerce platform limitations — all are theme/plugin configuration failures.
Highest CRO-impact issues
1. No "Buy Now" button — single-item purchases require 3+ steps. Abandonment driver.
2. Single payment gateway (Wompi) — unfamiliar to many Colombian buyers. MercadoPago would address this immediately.
3. Ambiguous product condition label — "Nuevo" conflates new/novelty. Direct purchase-intent signal damage for a used-book store.
4. No related products — leaves per-session revenue on the table for every single product page.
5. Missing analytics — HotJar, GA4 (not UA), Search Console not integrated. Running blind on conversion data.
Mobile experience failure
State management bug: "Ingresar o Crear Cuenta" shown in mobile dropdown even when user is logged in. Session-state is not propagating to mobile nav. Login/register on same page causes form confusion. Product images not adapting to mobile viewport. These are individually small but collectively destroy trust on mobile, which is likely the primary traffic source for a Colombian bookstore.
Critical: Plain-text password in registration email Critical
The registration email sends the user's raw password. This is an immediate indicator that either (a) WooCommerce's default email handling has been overridden by a plugin or custom code, or (b) passwords are being stored unhashed — a catastrophic GDPR and data security violation. Default WooCommerce does NOT expose plain-text passwords. This requires urgent investigation and likely points to a compromised or misconfigured email customization plugin. In Colombia, Law 1581 of 2012 (Habeas Data) applies.
Weak password policy
No minimum complexity enforced. Combined with the plain-text email transmission, this creates a severe credential exposure risk. WooCommerce's built-in password strength meter can be enforced — this requires only configuration, not development.
SSL certificate — adequate
SSL is present (Let's Encrypt via Hostinger). Not a risk item. Best Practices score of 95 confirms HTTPS is correctly implemented.
SEO equity — significant and at risk
SEO score of 85, RankMath configured, Open Graph present. ~4,000 indexed product URLs represent accumulated organic equity. Any platform migration carries risk of URL structure change + 301 redirect chain degradation. Google's crawl budget for the site would need to be carefully managed during any restructure. This is a strong argument against full rebuild to a new platform.
Used book inventory model — unique constraint
Single-copy SKUs (most books have qty: 1) means stock management is operationally critical. WooCommerce's native stock management handles this well. A migration to Shopify or another platform would require custom stock handling for single-copy items and cart reservation logic. The existing WooCommerce implementation already shows cart-reservation countdown timers — this is non-trivial feature parity to rebuild.
Colombian market context
Wompi is a legitimate Colombian payment gateway but has low consumer recognition. MercadoPago has dominant market penetration in Colombia. Adding MercadoPago (has WooCommerce plugin) is a configuration task, not a development task — high ROI, low effort. Wompi + MercadoPago covers the vast majority of Colombian e-commerce payment preferences.

Part V — Red-team attack
Challenges against recommendation B
Attack 1: "Replacing Elementor is essentially a rebuild — every page must be redesigned."
Assessment: Partially valid. Elementor replacement touches homepage, static pages, and potentially product archive pages. However, WooCommerce product pages are largely theme-controlled, not Elementor-controlled. With a WooCommerce-native theme (Kadence/Blocksy), the core commerce functionality remains intact. Estimated scope: 5–12 pages requiring Elementor redesign. This is not a rebuild — it is a focused layout migration. Attack does not falsify recommendation.

Attack 2: "Just add Cloudflare Pro + WP Rocket and the business will be fine at 60+ mobile score."
Assessment: Insufficient. Caching accelerates repeat visits but cannot reduce Elementor's first-load CSS parse time or the 3-carousel JS payload for new visitors (which include every marketing campaign target). The security issue (plain-text passwords) is unrelated to performance and requires intervention regardless. The 23 UX defects depress conversion independently of load time. Attack does not falsify recommendation.

Attack 3: "The team may lack the capacity to execute a restructure — optimization is lower risk to execute poorly."
Assessment: Operationally valid. A poorly-executed Elementor removal could break site functionality mid-project. This is the highest real risk. Mitigation: complete the restructure on a staging environment with full rollback capability before any production deployment. This risk is manageable but must be explicitly planned for. Attack partially modifies — adds risk management requirement but does not change verdict.

Revised position: Recommendation B stands. The operational risk in Attack 3 is addressed via phased rollout in the roadmap below.

Revised final verdict
Recommendation B — Partial Restructure — is confirmed after red-team review. The recommendation survives adversarial challenge. The root causes are implementation-layer (Elementor, budget hosting, carousel conflicts, no caching) and not platform-layer. WordPress + WooCommerce is capable of serving this business model at scale. The required interventions are targeted: hosting migration, page builder replacement, security remediation, and systematic UX repair. Estimated 3-year TCO advantage over rebuild: $12,000–35,000 in avoided development cost and SEO recovery risk. Expected post-restructure state: mobile score 70–80, LCP ≤2.8s, TBT ≤350ms, full UX defect resolution, conversion rate improvement measurable within 30 days of Phase 3.
